# xrayng APP UI 参考规范

这个文件用于指导 agent 设计标准化、产品化的 App UI，直接给出可落地的页面规范、配色关系和修改边界。

## 零、默认设计目标与北极星

这个 skill 默认把 xrayNG 的视觉目标理解为：

- 流畅：交互切换顺滑、状态反馈明确、列表与结构变化易读
- 轻盈：页面呼吸感足、留白干净、卡片不厚重、颜色不过度堆砌
- 现代：以克制、中性、系统化为主，不用营销式渐变和装饰性高光刷存在感
- 稳定：同类页面共享同一套 token、字阶、圆角、阴影和按钮体系
- 节能：不为了观感牺牲性能和功耗，不引入高频无效动画和复杂运行时绘制

### 这些词在本项目里的默认解释

- “现代”不等于高饱和、大渐变、毛玻璃泛滥、发光按钮、彩色正文
- “轻盈”不等于低对比、发灰、发虚、信息漂浮、缺少选中态
- “流畅”不等于到处做动效，也不等于用长时长转场制造顺滑错觉
- “高级感”不等于厚阴影、重描边或复杂装饰，而是统一、克制、稳定和干净
- “优化设计”默认同时包含可读性、可维护性、性能和功耗，不只是截图更好看

## 零点零五、Material 3 与 xrayNG 的关系

这个项目后续默认采用以下关系：

- `Material 3` 是组件系统、主题系统、语义 token、可访问性和系统适配的基底
- `xrayNG` 是产品视觉和页面语言层
- 设计目标不是“做成原生默认 MD3 demo”，也不是“完全手搓绕开 MD3”，而是“用 MD3 组件体系承载 xrayNG 的产品视觉”

### 默认实现策略

- Theme 继续建立在 `Theme.Material3.DayNight` 之上
- 颜色优先通过 `md_theme_*` 角色和语义 token 映射，不到处直接写 brand 颜色
- 卡片、分隔、开关、顶栏、按钮、进度、弹层优先使用 Material 组件或 theme-backed 实现
- 自定义 drawable 主要用于收 xrayNG 的风格层，不应用来替代本可直接使用的 Material 组件

### 对最新 MD3 方向的采用方式

- 采用：token 化、语义色、accessible contrast、edge-to-edge、自适应布局思路、更统一的 motion
- 谨慎采用：更强的 expressiveness、可变字重、形状变化、更多情绪化表达
- 不采用：过分 playful、夸张营销化、明显脱离工具型产品气质的视觉语言

这是结合最新公开 MD3 方向做出的产品化推断：xrayNG 应采用“克制版 MD3”，而不是完整 expressive 外放风格。

## 零点一、当前产品风格总览

在当前项目里，agent 不应把 UI 理解成“每个页面都可以各做各的”，而应先把现有产品语言看清楚：

- 当前产品主基调应切换为冷白灰背景、纯白面板、近黑文字、极少量中性强调
- 设置页、更多页、关于页、路由页属于同一类：整组 section panel、轻分隔、行内图标、不做图标底板
- 首页属于另一种更活跃的表达：更大留白、更圆卡片、更明显的胶囊、顶部栏目和底部 dock 更强
- 虽然首页更有结构感，但它和设置页仍共享相同的背景灰阶、文字气质和按钮克制度
- 整个产品不是靠厚阴影和强描边建立层次，而是靠背景深浅、圆角、间距、轻量分隔和少量高对比焦点

### 设置式页面当前观感

- 页面背景偏冷白灰，内容通常居中并限制最大宽度
- 一级内容按逻辑收进整组面板，不拆成很多独立卡片
- 行高稳定，图标尺寸稳定，图标与标题间距稳定
- 主次文字关系温和，说明文字较短，不靠大段说明堆层级
- 强调色只在主操作、可切换控件、关键状态中出现，默认优先中性深色

### 首页当前观感

- 首页比设置页更强调留白和节奏感
- 列表项是纯白卡片，圆角比设置页更大一些，但仍属于同一产品
- 协议标签、延迟标签、顶部分组、底部连接区都倾向胶囊化
- 顶栏和底部连接区是首页识别度的关键区域，改动时必须和列表项一起考虑
- 首页按钮虽然更显眼，但仍然保持克制，不依赖彩色高光和渐变装饰

### 对 agent 的直接要求

- 任何单页改版前，先判断它属于“设置式页面”还是“首页 / 列表式页面”
- 任何统一类任务，先看共享颜色、共享圆角、共享按钮和共享列表写法，再动单页
- 如果改完一个页面后，它和相邻页面不像同一个产品，说明方向错了
- 如果任务是“重构”而不是“重设计”，默认要求实现层变化对视觉效果近似透明

### 当前代码里已经固定下来的视觉规律

- 设置页、更多页、关于页、路由页通常有最大宽度约束，内容在中间，不直接铺满页面
- 设置行和更多入口的主行高基本落在 `68dp` 到 `72dp`
- 设置行图标多数是 `20dp`，图标与标题间距约 `14dp`
- 设置页分组通常是 `18dp` 圆角
- 首页列表卡片与底部 dock 的圆角明显更大，但仍维持冷白灰页面与纯白卡片的关系
- 首页按钮、tabs、协议标签、延迟标签普遍使用胶囊形态
- 路由页会在 section card 内再嵌 summary panel 和规则卡片，不是裸列表
- 日志页是“页面背景 + 主面板 + 面板内次级项”的层级，不是直接把日志 item 贴在页面底

这些规律应被视为现有标准，不应轻易打破。

### 当前分支的色彩增强落点

- 亮色模式统一使用冷白灰页面和纯白卡片，不再保留暖色、蓝色或黄绿色的装饰性渐变
- 设置页、更多页、关于页这类设置式页面优先使用纯白 section / row 与冷灰背景的关系，不把任何强调色扩散成面
- 首页列表卡片、dock 和 tabs 继续使用首页自己的 surface token，但这组 token应统一收敛到纯白与冷灰，而不是额外引入偏色
- 语义色只保留给延迟 badge、连接状态、成功 / 警告 / 失败反馈，不把绿 / 黄 / 红扩散成大面积装饰底色
- 首页主启动按钮优先用近黑或深石墨灰焦点，不使用彩色 glow 或营销式渐变
- 分组标题、section label 使用中性冷灰，不使用暖灰、蓝灰或装饰性偏色

## 零点五、严格限制 agent 的 UI 输出

为了确保统一，这个项目里的 agent 输出应被限制在现有产品语言之内：

- 允许：收颜色、收圆角、收间距、统一按钮、统一行高、统一列表项、统一表单
- 允许：在现有页面语言内部增强结构感，比如更清晰的 section、轻量 badge、统一空状态
- 允许：首页在保留当前语言的前提下做更强的留白、胶囊和顶部 / 底部联动
- 允许：通过共享 token、drawable、selector 和共享动效参数来统一体验
- 不允许：新增一整块 hero header、营销卡、大插画头图、大面积彩色背景
- 不允许：把设置页做成强装饰首页风格
- 不允许：把首页做成普通工具页或传统设置页
- 不允许：引入当前项目里不存在的新按钮体系、新卡片体系、新描边体系
- 不允许：同页混用多套圆角、多套按钮尺寸、多套文字层级
- 不允许：只把某一页做得更花，但和周围页面明显脱节
- 不允许：为了“现代化”引入高饱和主色、大面积彩色面、营销式渐变、长时间 shimmer、呼吸光效或彩色字体提示
- 不允许：为了“流畅”堆大量持续动画、滚动时高频透明度联动、复杂模糊或无法量化收益的视觉特效
- 不允许：为了“高级感”重新引入厚描边、重阴影、脏灰面板和多套并存的按钮底色
- 不允许：只优化视觉截图而忽略选中态、可读性、性能和功耗
- 不允许：能由 Material 组件承载的场景却退回到纯手工 view + 自定义背景模拟
- 不允许：把最新 MD3 误读成“默认紫色 + 更大圆角 + 更花的动画”
- 不允许：用户只要求“重构”或“改用 MD3 组件”时，顺手改掉既有卡片背景、边框、分隔线和色阶关系

### 输出前必须自检

- 这次改动是否仍然属于当前产品的两类页面语言之一
- 这次改动是否优先复用了共享颜色、共享卡片、共享按钮和共享行结构
- 这次改动是否减少了风格分裂，而不是制造了新的例外
- 这次改动如果截图给用户看，是否会被误认为另一个 App
- 如果这是一次“重构”，截图前后是否仍能读成同一个设计，只是实现更干净

## 零点七、视觉系统硬约束

如果用户没有明确指定其他方向，默认把下面这些当作硬约束执行：

### 色彩

- 基线始终是冷白灰页面背景、纯白或近白 surface、近黑主文字、冷灰次文字
- 语义色只用于真实状态：安全、延迟、连接状态、错误与成功反馈
- 正文、摘要、说明和 dock 动态提示默认保持中性色，不用绿 / 黄 / 红直接染正文
- 不把语义色扩散成整卡背景、整块 section 或大面积装饰

### 层次

- 层次优先依靠留白、卡片内外双层 surface、轻阴影、极弱高光和冷灰选中态建立
- 不依赖粗描边、厚阴影和装饰性 glow 建立层次
- 首页与设置页都要维持克制，不把首页做成花哨展示页，也不把设置页做成弱层次工具页

### 排版

- 主标题、正文、说明、badge 的字号体系保持少而稳，不频繁发明新档位
- 文本层级优先靠字号、字重、颜色和间距区分，不靠彩色或过度透明度
- 文字粗细优先跟随系统 font weight 设置，不在业务代码里新增分散、重复、不可控的字重方案

### 按钮与反馈

- 次级按钮优先透明点击面或弱底板，不用厚重灰底或彩色底
- 顶栏动作、列表更多、首页工具入口优先透明点击面 + 轻 ripple
- 反馈优先轻量、短时、明确；震动和动画都只保留关键动作

### Material 组件基底

- 设置页、编辑页、列表页、弹层的承载组件优先使用 `MaterialCardView`
- 分隔优先使用 `MaterialDivider`
- 布尔项优先使用 `MaterialSwitch`、`SwitchPreferenceCompat`
- 顶栏优先使用 `MaterialToolbar` 与 `AppBarLayout`
- 对话框和底部弹层优先使用 Material dialog / bottom sheet
- 若项目历史原因仍在使用 `EditText`、`Spinner`、`Preference`，也必须通过 theme / style / token 让其读感服从 MD3 体系
- 组件层升级到 Material 3 时，默认应保持既有视觉效果；若组件默认外观与既有设计冲突，优先调 style/token，而不是改变设计效果迁就组件默认值

## 零点八、动效、性能与功耗规范

### 动效原则

- 动效只服务信息变化、层级切换和状态反馈，不为装饰而存在
- 默认优先使用 `alpha + translation + scale` 这类低成本组合
- 动效时长应短且克制，避免拖沓；大多数过渡应落在短时长范围内
- 同一页面的动效节奏应一致，不混入多套完全不同的转场语言

### 严格限制

- 不做长期运行的 shimmer、呼吸光、循环浮动、连续 gradient 动画
- 不在滚动过程中持续高频改大量 view 的 alpha、scale、shadow 或 blur
- 不为了局部观感，把简单状态切换改成重型共享元素或复杂路径动画
- 如果一个动效无法提升可读性、识别度或操作反馈，就应视为冗余

### 性能与功耗

- 默认优先静态 drawable、selector、style、token，而不是运行时频繁拼颜色、阴影和形状
- 列表更新优先 payload 和局部刷新，不做整项反复重绑
- 长列表页面必须避免会引起频繁重绘的复杂背景和持续动画
- Chrome、dock、tabs 的透明或状态联动应尽量使用有限状态档位，避免连续 progress 导致无效 render
- 任何“更高级”的视觉方案，只要明显增加功耗、发热或掉帧风险，就不应默认采用

## 零点九、系统栏与布局方向

- 新的页面和重构页面默认按 edge-to-edge 思路设计
- 滚动页面的背景可以延展到系统栏区域，但关键内容要根据 inset 留安全区
- 顶栏如果不是实心粘连条，应提供与页面背景匹配的保护层或渐变过渡
- 不要叠加多层系统栏保护，也不要让保护层颜色和 pane 背景不匹配

## 二、设置式页面必须满足

- 页面头部轻量
- 不用大卡片头
- 不给图标加底板
- 只有主 CTA 使用深色焦点，不把蓝色当成默认强调色
- 普通行、分组面板、顶部入口不使用纯蓝底
- 不堆很多独立小白卡片
- 每个分组看起来像一个完整面板
- 危险操作只有真实存在时才展示

## 三、列表页必须满足

- 可以保留卡片结构
- 但圆角不能比设置页更大
- 操作按钮不能像另外一套组件
- 背景灰阶要与设置页一致
- 列表卡片允许纯白，但必须落在统一冷白灰页面体系内
- 夜间列表卡片不要做成纯黑底上浮一层亮块

## 四、首页黑字主题规格

- 首页背景：`#F3F6F9`
- 首页主卡片：`#FFFFFF`
- 首页次级胶囊 / tabs / 小按钮：`#FFFFFF`
- 首页主文字：`#121212`
- 首页次文字：`#7C8794`
- 首页描边：`#E2E8EF`
- 首页主按钮：近黑底 `#121212`，文字 / 图标使用 `#FFFFFF`
- 首页列表选中：优先用黑色小指示、轻描边或极弱提亮，不使用彩色选中块
- 首页空状态：弱图形 + 留白 + 2 到 3 个白色胶囊操作，不要大段说明
- 首页底部连接区：优先做成整块纯白 dock，黑色按钮作为焦点
- 首页底部 dock 卡片固定使用纯白背景，不混暖灰或脏灰；分隔线固定冷灰，优先接近 `#E6ECF3`
- 首页底部 dock 的文字层级固定为：状态 `10sp` 到 `11sp`、配置名 `17sp` 到 `18sp`、摘要 `12sp`
- 首页底部 dock 的摘要位默认显示辅助说明；在连接测试、出口信息等场景下，优先直接替换这条摘要，不额外堆第二条说明
- 首页顶部 tabs：外层容器和底部 dock 使用同一组纯白背景关系，选中态是容器内再包一层浅冷灰胶囊
- 首页工具按钮：顶部更多入口、加号按钮、列表更多按钮统一使用白色或透明点击面，不做彩色底板
- 首页搜索态：搜索输入框只保留一层白色输入胶囊，输入法弹出后键盘上方区域必须是页面本色，不能留下灰块遮挡
- 当用户要求去掉灰框时：首页顶部搜索、左侧 app 名称、加号和更多入口都改为透明点击面，反馈只靠轻触动画和无底板 ripple
- 当 tabs 并入顶部栏目时：优先做成文字 tabs + 黑色底部指示线，不再额外包浅灰大容器
- 当 tabs 与顶栏左右入口共处一行时：左右按钮和 app 名称要主动缩小，给中间分组让出空间，图标 / 文案必须居中对齐
- 首页顶栏背景可以改成从雾白到透明的纵向渐变，让列表与顶栏过渡更软，同时保留黑字主题的轻感
- 使用顶栏渐变时：`AppBarLayout` 承担背景层，`Toolbar` 自身保持透明，避免运行时代码把渐变覆盖掉
- 如果需要更接近全面屏，优先降低顶栏和底部 dock 的基础高度，并收掉它们与屏幕边缘之间多余的雾白留白
- 真正的首页 edge-to-edge 需要同时满足三件事：`WindowCompat.setDecorFitsSystemWindows(window, false)`、去掉布局里残留的 `fitsSystemWindows="true"`、手动把状态栏 inset 只加到顶栏容器而不是整页内容
- tabs 常驻时，滚动联动应改成“背景透明度变化”而不是“组件显隐变化”
- 顶栏透明联动：优先改 `AppBarLayout` 的背景 drawable alpha；底部 dock 透明联动：优先改 card background color alpha
- 真正的首页透明感通常要求顶栏覆盖在列表上方；如果顶栏还处在普通文档流里，就算背景透明，视觉上也仍会留出一整块空白
- 顶栏放进覆盖层后，绘制顺序要在列表之后，否则会被内容层级冲掉，透明效果也会显得失真
- 底部 dock 的列表底 padding 和自身底 margin 都要轻量处理；如果直接把 system bar inset 全量叠加，底部会出现明显空白，看起来像“白底托住卡片”
- 首页如果同时有顶栏、dock、搜索、IME、分组滚动联动，优先建立 `AppChromeStateReducer -> TopBarRenderer / BottomBarRenderer` 的链路，不要继续在 `Activity` 里分散直写 alpha
- “只在滑动时透明”的推荐语义是：`IDLE = STABLE`、`DRAGGING/SETTLING = SCROLLING_IMMERSIVE`、`SEARCH/IME = override state`、`EMPTY = EMPTY_STABLE`
- 顶栏 / 底栏透明度优先使用有限档位，例如 `solid / soft / floating`，避免连续 progress 造成高频无效 render
- Chrome 状态调试如果需要日志，只记录状态枚举、滚动相位、匿名 group 标识，不记录搜索词和节点信息
- 首页折叠态搜索按钮优先和右上角工具按钮统一成 `36dp` 量级的小圆按钮
- 首页左上如果不保留 app 名称：默认只放菜单按钮，尺寸可以比右侧工具按钮略大一档
- 首页右侧搜索、加号、更多按钮要统一图标视觉大小和相邻间距，避免出现系统默认搜索图标偏大、偏疏的问题
- 搜索和加号之间如果看起来过散，通常先处理折叠态搜索根视图的保留宽度，再处理加号的起始 margin
- 在不改搜索展开逻辑的前提下，可以把加号和更多合并为一个 action cell，减少右侧被 `ActionMenuView` 切成多个单元格的分散感
- 如果要进一步合并成单按钮，优先保留一个 toolbar hub 入口，通过底部菜单承载搜索 / 添加 / 更多动作
- hub 入口本身要避免继续长成普通“更多”图标，优先做成搜索主图标配纵向双点或提示点的小胶囊视觉
- 如果组合视觉和系统菜单残留同时存在，优先去掉残留搜索按钮和 overflow，再把 hub 收成单一图标，保证右侧只读出一个入口
- 子页面统一的优先级：先统一 `background -> section card -> form field -> icon action button` 四层基础界面关系，再改具体布局
- 设置页 / 工具页 / 资产页 / 日志页：优先使用居中的 section panel、18dp 圆角、低对比面板分层和 36dp 弱底板次级操作按钮
- 编辑页：背景统一为雾白页面，内容区优先分成 section card；输入框和 spinner 使用更浅一档 field 底色，不再直接贴在页面背景上
- 服务器协议编辑页：基础地址卡之外的协议字段优先继续分组进同层级 section card，避免出现“一个卡片 + 一串裸字段”的旧式结构
- 协议标签：优先近黑底白字；延迟标签：根据状态使用浅绿 / 浅米 / 浅红底，文字对比必须明显
- 首页启停按钮：默认纯黑圆按钮，进入 loading 后只保留圆形进度层，不允许看到矩形背景或裁切痕迹
- 如果首页夜间模式改成纯黑底，首页的 `color_home_background`、`bg_home_top_bar_gradient`、`bg_home_bottom_underlay_gradient` 必须成组更新，不能只改单个颜色资源

## 五、禁止项

- 禁止超过 `18dp` 的设置页分组圆角
- 禁止出现圆形图标底板
- 禁止重描边、重阴影
- 允许冷白灰背景上叠纯白卡片，但必须靠留白、轻分隔和统一边界建立层级，不能靠乱色修饰
- 禁止蓝 / 绿 / 黄装饰性高光、气泡光斑和无意义渐变
- 禁止为了“占位”增加不存在的功能入口

## 五点五、浮动面板 / 子菜单专项规格

- 这组规格适用于所有浮动菜单、子菜单、context menu、overflow panel 和轻量操作浮层
- 面板主体背景固定为纯白 `#FFFFFF`，不要混入暖灰主卡片色
- 如果需要毛玻璃观感，仍然以白色面板为主，只允许叠加轻 blur、连续高光和极弱轮廓，不把主体做成灰色雾面块
- 毛玻璃底色优先控制在白到极浅灰之间，最终读感必须仍是白色面板
- 毛玻璃高光必须连续覆盖整块面板，优先全高渐变或整块 foreground overlay；禁止使用固定高度的独立高光带
- 毛玻璃边缘只允许极弱描边和内侧高光，不允许读出明确硬边框
- 如果实现用了多层叠加，必须避免出现横向分层线、局部亮暗断层和错误参与布局测量的 overlay
- 主文字固定使用高对比近黑，优先 `#121212`
- 次要信息、未激活说明、toggle 轨道使用中灰，优先 `#8F949B`
- 行 hover / active / selected 统一通过浅灰圆角底块表达，背景优先 `#F0F1F3`，不要改文字主色
- 面板外圆角优先 `12dp` 到 `16dp`
- 面板内交互项的状态背景圆角优先 `8dp`
- 不使用明确实线边框；层级依赖白底、轻阴影和极弱轮廓，不能读成厚描边浮框
- 菜单项内边距优先上下 `8dp`、左右 `12dp`
- 左侧图标、正文、右侧控件之间保持固定 gap，所有左侧图标必须垂直对齐
- 分割线两端必须留白，不允许顶格切到容器边缘
- 有子菜单的项右侧必须显示右箭头
- 布尔项使用 toggle
- 单选项当前值右侧必须显示勾选
- 同一面板内优先使用单一的右侧状态语义：箭头对应子菜单、toggle 对应布尔值、勾选对应单选，不叠加重复提示

## 六、改版后的检查清单

- 页面背景是否统一
- 分组背景是否统一
- 圆角是否统一
- 分隔线是否足够轻
- 按钮是否收敛
- 图标是否直接放置
- 页面是否真的接近参考图
- 如果是浮动菜单：面板是否纯白、选中态是否为浅灰底块、右侧状态控件是否符合箭头 / toggle / 勾选语义
- 如果是毛玻璃浮层：高光是否连续、白底是否仍成立、是否完全没有上下分割线
