# V2rayNG APP UI 设计技能

## 作用

这个技能的目标不是讨论抽象概念，而是指导 agent 按统一标准产出像正式产品一样的 App UI。它直接用于：

- 修改现有页面视觉效果
- 统一不同页面的布局与层级
- 根据参考图重做 APP 页面
- 收紧圆角、留白、颜色、按钮样式
- 修正“同一应用里像两套界面”的问题
- 帮助 agent 在不同页面上持续做出一致、标准、可落地的 UI 决策

这个技能默认有一个强前提：

- 先理解当前产品已经形成的页面语言，再决定怎么改
- 先保证和现有页面属于同一个产品，再考虑加强视觉表现
- 如果现有项目已经有明确风格，agent 不应把单页改成另一套全新语言

## 使用方式

开始修改前先做三件事：

1. 查看参考规范：`references/design-standards.md`
2. 读取当前页面 XML / drawable / style，确认现状
3. 判断页面类型，再决定修改力度

如果任务涉及页面改版、风格统一、夜间模式、首页重构、设置页重画，额外必须做两件事：

4. 至少对照 2 个同级页面和 1 组共享样式 / 颜色资源，确认当前产品实际语言
5. 先用 4 到 6 句总结“这个项目现在长什么样”，再开始改

页面类型分为两类：

- 设置式页面
  - 更多
  - 设置
  - 关于
  - 备份与还原
  - 路由设置
  - 偏好页
- 列表 / 工具 / 首页页面
  - 节点列表
  - 订阅列表
  - 日志
  - 资产列表
  - 连接控制区

## 核心原则

### 0. 先继承，再优化

- 优先从现有产品语言里提炼规律，不要跳过观察阶段
- 优先复用共享背景、面板、按钮、列表、表单写法，不要让单页先长成新物种
- 如果首页和设置页风格强度不同，应理解为“同一产品里的两种页面语气”，不是允许随意混搭
- 没有用户明确要求时，不要为了追求新鲜感重做整套视觉

### 0.5 MD3 做底，产品语言做面

- 这个项目不是“原生 Material 3 demo”，而是“`Theme.Material3.DayNight` + 自定义产品层”
- 可以继续使用 Material 3 的 theme token、语义色、dialog、bottom sheet、基础可访问性规范
- 不要把页面直接改回标准 Material 3 组件观感；默认保留 v2rayNG 已形成的纯白底、灰阶面板、低装饰、整组面板语言
- 任何 UI 修改都优先理解为“在现有产品语言内统一”，而不是“提升 Material 3 纯度”
- 如果用户问“该不该改成 MD3”，默认结论应是：`MD3 做底更好，当前项目的产品层 UI 作为最终表现更好`

### 1. 先保证统一，再谈装饰

- 同一级页面的背景必须一致
- 同类型分组的圆角必须一致
- 同类按钮的尺寸、触感、颜色必须一致
- 不允许一个页面是“整组面板”，另一个页面还是“独立小白卡 + 小按钮”

### 2. 设置式页面严格贴近参考图

- 页面背景默认使用纯白 `#FFFFFF`，不要再引入暖灰底
- 内容按逻辑分组为整块面板，不要堆很多独立小卡片
- 图标直接放，不要额外图标底板
- 分隔线要轻，不能抢视觉
- 头部信息要轻，不要占太高
- 危险操作只有在真实存在时才出现，不能为了占位硬加一组
- 如果页面属于 `PreferenceFragment`，也要尽量把结果拉回“更多 / 关于 / 路由”同一套设置式页面语言，而不是接受系统默认 preference 观感

### 3. 列表页允许更强结构感，但不能脱离整体

- 首页、订阅、规则列表可以保留卡片化结构
- 但它们的圆角、背景灰阶、按钮处理必须与设置页属于同一套界面语言
- 操作按钮优先内嵌，不做突兀的小白块

## 当前项目实际风格

根据当前资源和页面实现，这个项目当前应统一收敛到下面这套产品语言，agent 在修改时应默认沿用：

- 整体基底是“纯白页面 + 浅灰面板 + 克制蓝色强调”，不是暖灰底、米色底或大面积彩色铺底
- 设置式页面强调整组面板感，内容在居中的 section card 里组织，分隔线轻，图标直接落在行内
- 首页是同一产品下更强的一套语言：更大留白、更圆的卡片、更明显的胶囊和底部 dock
- 首页列表项、顶部 tabs、底部连接区虽然比设置页更活跃，但仍然共享同一组白底、灰卡片和近黑文字
- 蓝色主要用于主操作、状态和选中反馈，不用于大面积铺底
- 文本层级偏克制：标题直接、说明短、次文字偏浅灰，不依赖大段说明制造信息层次
- 页面不是靠暖色氛围、重阴影和重描边撑层次，而是靠白灰背景深浅、圆角、留白和轻分隔

把这段理解错，后续 UI 往往就会跑偏。

## 现有组件家族

为了稳定产出，agent 应优先把页面理解成由几类已存在的组件家族构成，而不是每次重新发明：

- section card：设置页、更多页、关于页、路由页、日志页的大面板容器
- settings row / more action row：68dp 左右行高，20dp 图标，14dp 左右图标文字间距，轻分隔
- home list card：首页节点卡片，圆角更大，内部用协议胶囊、延迟标签、更多按钮组织
- home dock：首页底部连接区，和首页卡片是同一语言，不是独立的工具条
- summary panel：路由页顶部小统计面板，属于 section card 内的次级面
- subtle icon button：36dp 左右的弱底板次级操作按钮
- chip / badge：协议、订阅、状态、延迟等轻量信息标签
- preference intro / grouped preference row：设置页顶部说明和 `PreferenceFragment` 行项，需要拟合到 settings row / more action row 的同一套节奏里

如果需求能用这些现有组件家族解决，优先复用和微调，不要新造一套。

## 稳定产出流程

遇到 UI 任务时，推荐按这个顺序执行：

1. 先归类页面：设置页、列表页、工具页、首页、编辑页
2. 读取当前页、至少两个同级页，以及共享 `colors/styles/drawable`
3. 写出一段简短判断：当前页属于现有哪种页面语言，问题是“不统一”还是“本身不好看”
4. 先决定改动层级：只收颜色和圆角、调整结构、还是重做整个页面
5. 能先收共享资源时，先收共享资源；只有共享资源解决不了，再改单页结构
6. 修改后对照同级页面复看，确认没有长成另一套产品

如果 agent 跳过第 2 步和第 3 步，产出通常会不稳定。

## 严格输出边界

agent 在这个项目里做 UI 时，默认遵守下面这些硬约束：

- 不新增 hero 区、大横幅、大封面、大插画头部，除非用户明确要求
- 不把设置页改成首页式重胶囊列表
- 不把首页改回传统设置页或普通 RecyclerView 工具页
- 不引入新的主色体系；默认仍是纯白 / 浅灰 / 近黑 / 克制蓝
- 不把普通入口做成高饱和大色块按钮
- 不把单页改成高阴影、重描边、冷白卡片风格
- 不随意新增图标底板、彩色容器、装饰角标
- 不在同一页面里混用两套行高、两套圆角、两套按钮尺度
- 没有明确需求时，不重写整页信息架构，只做统一和收敛
- 如果共享样式已能解决问题，不允许绕开共享样式只改单页

## 页面级限制

### 设置式页面

- 默认背景保持纯白页面
- 默认用 section card 包内容，不拆成多张离散小卡
- 行内图标保持单色线性，不加额外底板
- section 间距、标题、分隔线、行高必须和已有页面一致
- “更多”页面是设置式页面的高优先级参考，不只是同级页面之一
- 设置页选项背景如果不统一，优先参考“更多”页的入口行背景和分隔结构，而不是另造一套 item background
- 对 `PreferenceFragment` 来说，外层分组背景和行内点击背景应分层：外层负责整组圆角与底色，行内负责 click/ripple
- 设置页顶部说明默认应存在，且应和 `About` / `Routing` / `Subscription` 子页保持相同轻量语气
- 设置页内容宽度优先使用 `settings_page_max_width`，不要直接沿用首页或普通列表页的宽度约束

### 首页 / 列表页

- 保留首页的留白、胶囊、dock、协议标签、延迟标签这套语言
- 首页圆角可以略大于设置页，但不能继续无限放大
- 首页工具按钮默认和现有顶部入口、列表更多按钮属于同一尺寸体系
- 首页底部连接区必须视为首页结构的一部分，不能临时变成独立工具栏

### 工具页 / 日志页 / 路由页

- 仍然属于设置页大语言，不要单独做成第三套视觉
- 可比设置页更直接，但容器、圆角、背景深浅、次级按钮仍要复用现有写法
- 路由页和日志页的列表项默认应收在统一面板里，而不是裸贴页面底

### Preference 页面专项规则

- `PreferenceCategory` 标题、顶部说明、设置行，不应保留系统默认密度和默认组件观感
- 设置行标题默认靠近 `15sp` 到 `16sp`，副标题默认靠近 `12sp`
- `Preference` 行的分隔线、图标大小、文字起始位应与 `MoreAction.Row` 对齐
- 若使用自定义 adapter 给 `PreferenceFragment` 分组，必须显式排除顶部 intro / 非行项，避免错误套用首尾圆角
- 不要同时让外层背景和行背景都带完整 ripple，否则按压时会出现双层反馈或背景不统一

### 浮动面板 / 菜单专项规则

- 这组规则适用于 overflow menu、上下文菜单、底部更多菜单中的二级面板、带子菜单的浮层、右键式操作面板和轻量设置弹层
- 浮动面板背景默认使用纯白 `#FFFFFF`，优先强调“与底层内容隔离”，不要继续沿用首页 dock 或设置页暖灰主卡片底色
- 如果用户明确要求毛玻璃效果，默认理解为“白色主底 + 轻 blur + 连续高光”，不是把面板改成灰色半透明块
- 毛玻璃面板的底色仍应读作白色，允许使用接近 `#FFFFFF` 的低透明白或白到极浅灰的连续渐变，但最终观感必须仍是白面板
- 毛玻璃高光必须是整块连续过渡，优先使用全高渐变或整块前景 overlay；禁止在中上部再叠一条固定高度的 sheen 条带，否则很容易出现上下分割线
- 毛玻璃边缘只允许极弱外轮廓和内高光，优先接近白灰描边；禁止明显实线边框
- 毛玻璃层级主要依赖轻微 blur、顶部连续高光、极弱轮廓和柔和阴影共同建立；不要靠重阴影或多层切块假装玻璃
- 如果实现使用多层 drawable / overlay，必须确保所有层都不参与错误测量、不制造横向切线、不在卡片中下部形成明显亮暗分界
- 主要文本使用接近纯黑的高对比颜色，优先接近 `#121212` 到 `#1F1F1F`
- 次要信息、未激活文案、说明、toggle 轨道、辅助状态统一降到中灰，优先接近 `#8B8F97` 到 `#9AA0A6`
- hover / active / selected 不通过改文字颜色表达，而是给整行叠加浅灰圆角背景，优先接近 `#F0F1F3`
- 浮动面板外圆角必须明显且统一，优先 `12dp` 到 `16dp`
- 面板内部可点击项的 hover / pressed 背景圆角应更小，优先 `8dp`
- 浮动面板不使用硬边框；边界主要靠轻阴影和极弱轮廓建立层级，如果必须补轮廓，只允许用接近白灰的低对比描边，不能读成明确描边盒子
- 菜单项内部边距保持统一，优先上下 `8dp`、左右 `12dp`，保证点击区足够大
- 图标、标题、右侧状态控件必须使用固定间距，并让所有左侧图标共用同一条垂直对齐线
- 分割线不能顶格，前后要保留均匀留白，两端都要留 margin
- 包含子菜单的行，最右侧必须使用右向箭头 `>` 或等价 chevron 表达层级
- 布尔开关项必须使用 toggle，不允许用文案“开 / 关”硬写状态
- 单选列表项必须在当前生效项右侧显示勾选标记 `✓` 或等价 check icon
- 如果同一浮层里同时出现子菜单、toggle、单选项，右侧状态表达必须遵守统一优先级：子菜单显示箭头、布尔项显示 toggle、单选项显示 check，不混搭重复提示

## 白底灰卡约束

- light theme 默认页面背景必须使用纯白 `#FFFFFF`
- 所有主要卡片、分组面板、dock 和次级容器优先使用浅灰色阶，而不是米色、奶油色或暖灰色
- 主要文本使用接近纯黑的高对比度颜色，优先接近 `#121212` 到 `#1F1F1F`
- 次要文本、未激活状态、说明文本统一使用中灰，优先接近 `#8B8F97` 到 `#9AA0A6`
- hover / active / selected 不通过改文字颜色表达，而是通过浅灰圆角背景块表达，优先接近 `#F0F1F3`
- 浮动面板和菜单主体必须使用纯白底；内部可点击项的反馈背景使用浅灰 `#F0F1F3`
- 所有浮动面板外圆角优先 `12dp` 到 `16dp`，内部交互反馈圆角优先 `8dp`
- 浮动面板不使用明显实线边框；层级由轻阴影、极弱外轮廓和白灰底色建立
- 菜单项内部 padding 保持统一，优先上下 `8dp`、左右 `12dp`
- 所有左侧图标在同一垂直线对齐；图标、标题、右侧控件之间保持固定间距
- 包含子菜单的项右侧必须显示右向箭头；布尔项使用 toggle；单选项右侧显示勾选标记

## 建议交付格式

当任务是页面优化、风格统一、夜间模式重构、首页改版时，agent 在内部应尽量按这个格式组织判断和实施：

1. 当前页面属于哪一类页面语言
2. 当前页面和同级页面相比，最不统一的 3 个点是什么
3. 这次改动是收敛统一，还是确实需要提升风格强度
4. 哪些改动应该先落在共享样式 / 共享颜色 / 共享按钮上
5. 哪些改动只能落在当前页面
6. 改完以后，和哪两个相邻页面重新对照确认

如果任务较大，agent 应优先产出“风格判断 + 改动边界”，再开始写代码。

## 严格规格

### 页面背景

- 新参考图属于“雾白页面 + 浅灰卡片”的低对比层级，不是厚重暖米色底
- 亮色模式页面底：优先更亮的雾白，接近 `#F7F6F2`
- 亮色模式一级卡片 / 主面板：比页面略深一档，接近 `#F1EFEC`
- 亮色模式设置组背景 / 弱层级面板：再深一档，接近 `#ECE9E4`
- 分隔线：接近 `#DFDAD3` 到 `#E6E1DA`
- 夜间模式背景不要纯黑，优先深灰蓝，接近 `#11151B`
- 夜间模式一级卡片：接近 `#1A2027`
- 夜间模式弱层级面板：接近 `#232B34`
- 如果用户明确要求“夜间背景纯黑”，允许把页面背景切到 `#000000`
- 启用纯黑夜间模式后，必须同步把主卡片、弱层级面板、输入框底、状态 badge 和描边整体压暗一档；不能只把背景改黑
- 启用纯黑夜间模式后，夜间渐变 drawable 也必须同步改成黑色过渡，避免顶栏 / 底部 dock 残留深灰蓝色带

### 圆角

- 设置页分组圆角：`18dp`
- 设置页行点击背景圆角：`18dp`
- 首页 / 列表项圆角：`16dp` 到 `18dp`
- 小按钮圆角：`16dp`
- 禁止出现 `24dp`、`28dp` 这一类偏夸张大圆角，除非用户明确要求

### 间距

- 页面左右内边距：`16dp`
- 页面顶部正文起始间距：`8dp` 到 `12dp`
- 分组标题与上一组间距：`20dp`
- 行最小高度：`68dp`
- 行左右内边距：`16dp`
- 行上下内边距：`12dp`
- 图标与文字间距：`14dp`
- 分隔线左缩进：`52dp`

### 文本

- 行标题：`15sp` 到 `16sp`，加粗
- 行副标题：`12sp` 到 `13sp`
- 分组标题：弱化显示，不抢正文
- 页面说明：短句即可，不要写成长段

### 图标与按钮

- 设置式页面图标统一为单色线性图标
- 不使用图标底板
- 主 CTA 按钮允许使用蓝色，但只使用 `#2785FF`
- 除主 CTA 外，不把蓝色铺满到普通列表行、分组卡片、顶部入口按钮
- 右侧操作按钮优先做成无底板或弱底板的内嵌操作

## 修改时必须检查

- 有没有多余的图标底板
- 有没有过大的圆角
- 有没有无意义描边
- 有没有偏白、偏蓝、偏冷的独立卡片跳出来
- 有没有把一个分组错误拆成多个卡片
- 有没有加了并不存在的功能入口
- 有没有只把单页做漂亮，却和周围页面不再属于同一产品
- 有没有在设置页误用首页的重胶囊语言，或在首页误退回老式设置页布局
- 有没有绕开共享样式直接在单页硬写一套新颜色 / 新圆角 / 新按钮
- 有没有发明出当前项目里根本不存在的新组件类型
- 如果是设置页 / 偏好页：有没有和“更多”页出现不同的行背景逻辑、不同的分隔线缩进、不同的标题密度
- 如果是 `PreferenceFragment`：有没有漏掉 intro 项、category 项、expandable 项的特殊处理
- 如果是浮动面板 / 菜单：有没有误用暖灰主卡片底、有没有把选中态做成改字色、有没有漏掉子菜单箭头 / toggle / check 标记
- 如果是毛玻璃菜单：有没有把白底做成发灰半透明块、有没有叠出固定高度高光条、有没有产生肉眼可见的上下分割线

## 常见处理方式

### 当页面是“更多 / 设置 / 关于”

- 直接按整组面板重做
- 去掉头部大卡片
- 去掉图标底板
- 去掉无意义箭头或徽标
- 把危险项独立，但仅在功能真实存在时保留
- 如果是设置页且基于 `PreferenceFragment`，优先做三件事：
  1. 补顶部说明
  2. 统一 category / item layout 到设置式页面规格
  3. 用共享 row background 对齐“更多”页，而不是接受默认 preference item 背景

### 当页面是“首页配置列表 / 订阅列表”

- 保留列表感
- 收小圆角
- 降低按钮存在感
- 保持整体灰阶接近设置页

### 当页面是“首页黑字主题重构”

- 首页整体参考浅底大留白界面，黑色负责文字、图标、选中态和主操作，不是页面背景
- 页面底优先接近 `#FBFAF7`
- 主卡片 / 底部 dock 优先接近 `#F2EFEB` 到 `#F4F1EC`
- 次级胶囊 / tabs / 小按钮优先接近 `#EEEAE3`
- 主文字优先接近 `#121212`
- 次文字优先接近 `#8C877F`
- 主按钮优先用纯黑或近黑，接近 `#121212`
- 主按钮文字 / 图标优先白色
- 配色保持克制，除状态反馈外不要再引入明显蓝色、绿色大面积块
- 空状态尽量留白，不堆说明卡；操作入口改成少量浅灰胶囊按钮
- 底部主控制区可以做成一整块浅灰 dock，背景要明显比页面更深半档，黑色按钮作为唯一强对比焦点
- 顶部 tabs 和底部 dock 要共用同一类圆角、灰阶和选中逻辑，看起来像同一套浅灰胶囊组件
- 首页顶部更多入口、加号按钮、列表里的更多按钮，也要共用同一套浅灰胶囊按钮语言，不要回退成透明图标按钮
- 搜索态展开后，工具栏除搜索输入本身外必须保持透明收口；输入法弹出时，键盘上方不能残留灰色面板、dock 残影或延迟隐藏的遮挡层
- 如果用户明确不要灰色框，顶部搜索、左侧 app 名称、加号和更多按钮要回到透明点击面，只保留图标 / 文字本体和无底板反馈
- 订阅分组 tabs 可以整体并入顶部栏目，不一定单独悬浮成一块；此时优先用文字 + 下划线 / 底部指示来表达选中态
- 当 tabs 放进顶栏同一行时，左右工具入口必须同步缩小一档，图标和文字保持垂直居中，避免挤压中间分组
- 首页顶栏可以使用“上实下透”的浅色渐变透明背景，让列表更贴近顶栏但不显得生硬分层
- 顶栏如果使用渐变透明背景，代码里不要再把 `Toolbar` 单独刷成纯色，否则会直接盖掉渐变层
- 如果用户要求更强的全面屏效果，优先先压低首页顶栏和底部连接卡的基础高度，再减少它们上下多余的底色留白
- 如果用户要求首页真正 edge-to-edge，必须关闭 `decorFitsSystemWindows`，并去掉布局里遗留的 `fitsSystemWindows="true"`；否则顶部只会是假透明
- 首页分组 tabs 如果用户明确要求常驻，滚动时不要再隐藏；此时只联动顶栏背景透明度，不要把 tabs 一起淡掉或收起
- 顶栏滚动透明效果优先只改背景渐变层 alpha，不要直接改 `Toolbar` 整体 alpha，否则图标和文字会一起发虚
- 底部连接 dock 的透明效果优先只改卡片底色 alpha，不要直接把整块 view 降 alpha，否则按钮和文案会一起变淡
- 如果首页顶栏看起来仍然像一整块留白，优先把顶栏从普通纵向流布局改成覆盖在列表上方的叠层，再配合列表顶 padding 控制首屏间距
- 顶栏作为覆盖层时，在 `CoordinatorLayout` / `FrameLayout` 里的绘制顺序必须放在列表之后，否则即使透明，视觉上也不像真正浮在内容上面
- 首页列表如果要滑进状态栏和底部 dock 区域，优先降低列表本身的顶部 / 底部 padding，而不是一味继续降低透明度
- 底部 dock 的透明效果要克制，默认态应接近实色，只在上滑列表时轻微变透；不能把它做成发飘的大白块
- 首页 chrome 如果同时涉及滚动、搜索、IME、分组切换，不要继续在 `MainActivity` 里堆 `if/when`；优先抽成一个显式的 `AppChromeStateReducer`
- `AppChromeStateReducer` 只接收事件和纯状态，不持有 `View`；顶栏和底栏分别由 renderer/controller 渲染，`Activity` 只做接线
- “只在列表滑动时透明”必须基于显式的 `scrollState`（`DRAGGING / SETTLING / IDLE`）而不是只看 `dy`，否则停止滚动后一定会残留错误透明态
- reducer 的输出优先做离散状态 + 有限透明档位，并在输出层做 `distinctUntilChanged`；不要让每 1px 滚动都生成一套新视觉状态
- 如果首页 chrome 状态系统需要调试，只允许在 debug 环境记录匿名状态转移，不记录分组名、节点名、搜索词等敏感内容
- 折叠态搜索按钮也要纳入顶栏同一套尺寸规则，优先和加号 / 更多保持一致的轻圆按钮尺寸
- 当用户要求删掉首页左上 app 名称时，默认态只保留菜单入口；如果需要反馈文案，只在短暂状态提示时再显示
- 顶栏右侧搜索、加号、更多三枚按钮必须保持相同图标视觉尺寸和更紧的节奏，不能一个大一个小、一个疏一个密
- 如果右侧搜索和加号仍显得分散，优先检查折叠态 `SearchView` 根节点占位宽度，而不是只改相邻按钮 margin
- 如果不想重写搜索展开链路，优先把加号和更多并成一个 toolbar action cell，先收紧右侧次级操作
- 如果用户明确要求把搜索、加号、更多合成一个入口，优先保留一个单独的 toolbar hub 按钮，其余动作收进底部菜单
- 单独的 toolbar hub 按钮不要再画成普通“更多”按钮，优先做成更明确的组合入口，例如搜索主图标配纵向双点或提示点的小胶囊
- 如果组合入口在窄顶栏里显得混乱，优先退回单一、可解释的图标，并把原搜索项和 overflow 菜单都隐藏掉，避免三套入口同时出现
- 协议标签和延迟标签必须比卡片底更清晰，允许用近黑协议胶囊、浅绿 / 浅米 / 浅红延迟标签来提升可读性
- 主操作按钮在 loading / 启停过渡时必须保持纯圆叠层，只显示圆形进度，不允许出现矩形底框或多余描边

### 当页面是“工具页 / 日志页”

- 结构可以更直接
- 但背景、面板、圆角和文字层级仍然要统一

## 完成后的验收

- 同一应用内所有新改页面看起来像一个产品
- 更多页、设置页、关于页应尽量接近参考图
- 圆角不能再偏大
- 颜色关系必须稳定
- 编译必须通过
- agent 应能用简短语言说明：当前页面沿用了项目已有哪套页面语言，改动后为什么更统一
- 如果修改了设置页 / preference 页，验收时必须额外回答：
  1. 这页的行背景是否已经与“更多”页同类项统一
  2. 外层分组背景与行内点击背景是否明确分层
  3. 顶部说明、标题密度、分隔线缩进是否与其他设置式页面对齐

## 当前项目配色沉淀

- 当前项目统一使用“雾白页面 + 浅云灰卡片 + 克制蓝色强调”
- 新参考图的关键关系是：亮色模式下页面背景比卡片更亮，卡片不要再做成奶白上浮块
- `md_theme_background` 作为全局页面底色，使用接近 `#F7F6F2` 的高亮度中性暖白
- `md_theme_surface` 作为默认卡片底色，使用接近 `#F1EFEC` 的浅灰白，控制和背景的对比
- `md_theme_surfaceVariant` 作为设置分组、弱化面板、次级容器底色，使用接近 `#ECE9E4` 的更深一档雾灰
- `md_theme_primary` 只用于主操作、选中态和关键状态，不铺到整页容器
- 夜间模式保持同一层级关系：背景 < 主卡片 < 弱层级面板，不使用纯黑 + 亮白的割裂组合
- 如果用户明确要求纯黑夜间模式，允许把夜间背景落到 `#000000`，但必须同时把 `md_theme_surface` / `md_theme_surfaceVariant` 或首页的 `color_home_*` 面板色压暗，维持“背景 < 主卡片 < 弱层级面板”的层级
- 纯黑夜间模式下，首页优先同步检查 `color_home_background`、`bg_home_top_bar_gradient`、`bg_home_bottom_underlay_gradient`，子页面优先同步检查 `md_theme_background`、section card、form field、outline
- 首页黑字主题允许使用单独的 `color_home_*` 资源，不强行复用全局页面配色
- 子页面统一时，优先先收三层共用界面基础：页面背景 / section card / 表单 field，再去改单页结构
- 设置页、工具页、资产页、日志页的次级图标按钮要统一成同一套 `36dp` 弱底板按钮，不再混用 borderless 点击区
- 日志、订阅、路由、资产等列表页如果还在直接贴页面底，优先补一层低对比卡片容器或列表卡片间距，让它们和设置页同属一套面板语言
- 服务器、订阅、路由编辑页即使不完全重画，也至少要统一背景、顶部留白、section card 和表单字段底色，先消除“老式满屏表单”观感
- 服务器编辑页里，`layout_address_port` / `layout_transport` / `layout_tls` 之外的协议专属字段，也要收进独立 section card；不要让用户名、密码、flow、WireGuard 密钥这类字段直接裸露在页面背景上
